96. Accelerating Screening of 3D Protein Data with a Graph Theoretical Approach


Cornelius Frömmel1, Andrean Goede1 , Clemens Gröpl2, Stefan Hougardy2 , Till Nierhoff2 3, Robert Preissner1 , Martin Thimm2



Keywords: superposition, data analysis, data mining, DIP

Introduction

The Dictionary of Interfaces in Proteins (DIP) is a data base containing information about interacting parts of the three-dimensional structure in proteins that are called patches [1]. A patch is a set of atoms of a secondary structural element (SSE) that are in spatial proximity with atoms of another structure (SSE, solvent, or ligand). The spatial similarity between two patches can be expressed in terms of a 3D similarity score. Given two patches, this score is computed by choosing subsets of the atoms in both and then finding a rigid motion that minimizes the root mean square distance (rms) of the chosen atoms. The score value is then

rms/Å
where is the number of chosen atoms divided by the size of the smaller patch. This score acts as an assessment criterion for the prediction of binding sites or protein folding. Currently, only a heuristic value is computed, because the problem is hard.

DIP is a growing database that currently contains more than patches of several thousands of proteins. Its purpose is to serve as a repository in which patches similar to given query patches can be found. However, even the computation of a heuristic value for one similarity score takes about one tenth of a second. As a consequence, a comparison of a query patch to all patches in DIP already at present takes more than one day. With the growth of DIP more efficient search strategies are therefore necessary. In this work we address two questions:

  1. Given a query patch, how similar are other patches in terms of the score that are relevant to the query.
  2. Given a query patch, is there a way to search for the relevant patches, scanning only a small part of the database.

Results

As to the first question, it turns out that there are essentially three ranges of the similarity score which can be attributed with (i) noise, (ii) accidental similarity, and (iii) homology, respectively. We consider the similarity graph, i.e. the graph whose vertices correspond to patches and which has an edge between two patches if and only if their similarity score exceeds a certain threshold value. We find that the size of the largest connected component of the similarity graph, plotted against the threshold value, has two remarkable bends at the values which separate the three ranges described above. In contrast to this, the same plot has no bends at all if the score values are randomly permuted among the patch pairs (Figure 1). We infer that the similarity data has structural properties that might be used to address the second question.

We have tested several approaches to reduce the number of necessary comparisons for a given query patch. They have been measured in terms of a speed up factor that is defined as follows: If the number of patches in the data set is , then the speed up factor is divided by the number of comparisons needed by the procedure to identify those entries that have a similarity score with the query in one of the ranges (ii) or (iii). So, the speed up factor measures how many times faster than linear search the tested procedures are.

For the range (iii), corresponding to homologous patch pairs, a hierarchical organization of the data leads to a speed up factor of the order of magnitude of . Range (ii) is especially interesting, because the patch pairs in this range that are not homologous nevertheless share some structural features. These exceptionally similar pairs can provide interesting information e.g. about ancestral relationships and can only be found using folding information about the proteins. We found that this range is more difficult to handle and that substantial acceleration can only be achieved at the expense of false negatives. We have developed a search structure that is efficiently computable and that could be tuned to find 85% of the relevant patches with a speed up of ten or 95% with a speed up of four (Figure 2).
