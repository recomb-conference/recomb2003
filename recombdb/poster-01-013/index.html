13.  A Service-Centric Computing Environment for Heterogeneous Biological Databases and Methods

Uwe Radetzki[1],  Thomas Bode1,  Gabriele Witterstein1,
 Melanie Gnasa1,  Armin B. Cremers1

Keywords: web services, mediators, ontologies, heterogeneity, service integration

1     Introduction

On this poster we present new resources and concepts, and how we apply them to build a service-centric computing environment supporting life science applications. The main concepts used are web services, mediators, and ontologies. Today, a large number of different incompatible data sources and distributed methods are available via the world wide web. But even if they contain similar information or provide similar algorithms they differ in data types, request semantics, formats, and structures. An example of similar databases with different structures and formats include the various protein signature databases which are developed to address different aspects of functional classification of proteins [1]. Further, the rapid pace of change in this field – new databases and algorithms emerge, new species are sequenced and annotated, etc. – reduces the deployment of stable standards and formats [3]. Thus, we have to answer a very heterogeneous and dynamic situation.

A typical functional genomic analysis today can look like this: formulating a request for a specialized genome database in order to retrieve the gene which is unique to an expressed sequence tag (EST). The output has to be converted by oneself to serve as new input depending on the next tool. For instance, the user has to formulate a BLAST-X query into one or more different protein databases in order to get the homologous protein sequences. Based on the best hits further information retrieval sessions start (e.g. publication search), and so on. Remember, the whole process is denoted by data and method sources which have different data formats and service specifications that have to be taken into account.

Thus, what we need is a flexible and adaptable software solution which can keep up with these moving objectives. It has to provide abilities to include new or changed data sources as well as distributed methods simultaneously, and to combine them in a meaningful manner. Further requirements are: supporting standards as much as possible, uniform access to distributed data and method services, easy to handle for users, supporting different user and project demands, and automation of repeated analysis steps.

2     The IRIS Approach

In the IRIS platform (Interoperability and Reusability of Internet Services) we are applying recent concepts in the field of software technology in order to build a flexible service-centric computing environment [2]. These sophisticated concepts comprise web services, ontologies, and mediators. An overriding goal of all aspects of this project is that the user will be supported as much as possible in building new service compositions. Here, we try to keep in mind the level of computer training and knowledge of the most biologists that will be using this software platform. In general, we are convinced that in the near future every web-enabled data or method service is accessible through standardized protocols based on XML and HTTP [5], e.g. see the XEMBL service [4].

In IRIS the user can describe new service compositions in a declarative manner. For that he will use a special XML-based workflow language where the data and control flows have to be defined. In this connection, an open question is the incompatibility of data types and request semantics supported by the different services. For this purpose we developed a specialized mediator approach. A mediator supports a specific data conversion which the user can incorporate into a given workflow. It is possible to compose several mediators. This allows users to reuse existing mediators. We assume that by-and-by the number of available mediators grows, and the user not only has to develop new mediators but also can download required mediators from the net. In the later case, the mediator has only to be adapted adequately. In order to find the required mediators we will use ontology-based search mechanisms, but this is work in progress. Note, every service aggregation defined in IRIS will be published as new web service. In this way, IRIS supports a hierarchically composition model. Figure 1 illustrates an example of a service aggregation in IRIS.

Beyond, we are thinking of IRIS mediators which can query well-known and widely accepted ontologies (e.g. GenOntology) independently in order to allow semi-automatic customization of supported data formats. Mediators involved in such negotiation process have to use the same ontologies. In this context the main objective will be to support the user with potential data conversions which finally have to be reviewed by the user. This smart mediation is future work.
